# 상품 리뷰 · 상품 문의(Q&A) 기능 설계

## Context

`supabase/products.sql`에 `product_reviews`, `product_qna` 테이블과 RLS 정책, 통계 자동 갱신 트리거(`update_product_review_stats`)까지 이미 정의되어 있고, `src/hooks/useProducts.ts`에는 `fetchProductReviews`/`fetchProductQna` 읽기 전용 조회 함수가, `src/app/shop/[category]/[id]/page.tsx`에는 리뷰·Q&A 탭 UI(리스트 렌더링)까지 이미 구현되어 있다. 다만 **"리뷰 작성하기"/"문의하기" 버튼이 onClick 핸들러 없이 방치된 스텁**이라 실제 작성(INSERT)이 불가능하고, 어드민에서 리뷰를 숨기거나 문의에 답변하는 관리 화면도 전혀 없다.

사용자 결정 사항:
- **리뷰 작성 진입점은 상품 상세 페이지가 아니라 `settings/purchases` (구매내역)** — 상품 상세 페이지는 리뷰 "조회"만 담당한다. 리뷰는 실제로 구매를 완료한 주문 항목에서만 남길 수 있어야 하므로, 개별 상품(`product_id`)을 갖고 있는 `src/app/settings/purchases/[id]/page.tsx`(주문 상세, `order.order_items` 배열)에 상품별 "리뷰 작성" 버튼을 추가한다.
- **리뷰는 상품당 1개가 아니라 "주문(구매 건)당 1개"** — 같은 상품을 여러 번 구매했다면 구매할 때마다 별도로 리뷰를 남길 수 있다. "주문 상품" 섹션에 버튼이 있는 게 오히려 이 모델과 정확히 맞는 이유. 이를 위해 `product_reviews`에 `order_item_id` 컬럼을 추가해 리뷰를 특정 `order_items` 행(=특정 구매 건)에 귀속시킨다.
- **Q&A(문의)는 원래 설계대로 상품 상세 페이지에서 작성** — 구매 여부와 무관하게 로그인만 하면 작성 가능.
- **리뷰에 사진 업로드 포함** — `product_reviews.images JSONB` 컬럼을 실제로 채운다. 커뮤니티 글쓰기(`src/app/community/write/page.tsx`)의 Supabase Storage 업로드 패턴(`totodo_pub_storage` 버킷)을 재사용.

## 1. DB 마이그레이션 — `supabase/products.sql` 하단 마이그레이션 섹션에 추가

### 1-1. 리뷰를 특정 구매 건에 귀속 (`order_item_id`)

리뷰를 "주문당(=order_item당) 1개"로 제한하려면 어떤 `order_items` 행에서 나온 리뷰인지 알아야 한다. `product_reviews`는 현재 `product_id`/`user_id`만 갖고 있어 같은 상품을 재구매해도 리뷰를 구분할 방법이 없다.

```sql
ALTER TABLE product_reviews
  ADD COLUMN IF NOT EXISTS order_item_id INTEGER REFERENCES order_items(id) ON DELETE SET NULL;

-- 구매 건(order_item) 하나당 리뷰 하나만 허용
CREATE UNIQUE INDEX IF NOT EXISTS idx_product_reviews_order_item_unique
  ON product_reviews(order_item_id)
  WHERE order_item_id IS NOT NULL;
```

`ON DELETE SET NULL`로 둬서, 주문이 나중에 삭제되더라도 이미 작성된 리뷰 자체는 orphan 상태로 남아 사라지지 않게 한다. 매장/어드민 흐름에서는 항상 `order_item_id`를 채워서 insert하도록 애플리케이션 레벨에서 강제한다 (DB `NOT NULL`은 걸지 않음 — 향후 관리자가 직접 리뷰를 대신 등록하는 등의 예외 케이스를 막지 않기 위함).

### 1-2. 관리자 RLS 정책

현재 `product_reviews`/`product_qna`는 소유자(`auth.uid() = user_id`) `FOR ALL` 정책만 있고 관리자용 정책이 없다 (다른 테이블은 `products_select_admin`처럼 `public.is_admin()` 기반 정책을 갖는 패턴을 따른다).

```sql
-- product_reviews: 관리자 조회(숨김 리뷰 포함)/수정(숨김처리)/삭제
CREATE POLICY "product_reviews_select_admin" ON product_reviews FOR SELECT USING (public.is_admin());
CREATE POLICY "product_reviews_update_admin" ON product_reviews FOR UPDATE USING (public.is_admin());
CREATE POLICY "product_reviews_delete_admin" ON product_reviews FOR DELETE USING (public.is_admin());

-- product_qna: 관리자 조회(비공개 포함)/답변 작성/삭제
CREATE POLICY "product_qna_select_admin" ON product_qna FOR SELECT USING (public.is_admin());
CREATE POLICY "product_qna_update_admin" ON product_qna FOR UPDATE USING (public.is_admin());
CREATE POLICY "product_qna_delete_admin" ON product_qna FOR DELETE USING (public.is_admin());
```

이 정책이 없으면 어드민이 숨겨진 리뷰나 비공개 문의를 조회/답변/삭제할 수 없다 (RLS가 막음).

## 2. 유저 훅

### 2-1. `src/hooks/useMyProductReview.ts` (신규)

`src/hooks/useMyLectureReview.ts` 패턴을 그대로 따르되, 조회/등록 키가 `product_id`가 아니라 **`order_item_id`** (특정 구매 건)이다. 이 훅은 구매가 이미 검증된 컨텍스트(주문 상세 페이지의 리뷰 모달)에서만 호출되므로, 별도의 구매 이력 재검증 없이 `is_verified_purchase: true`를 고정한다.

- `fetchMyReview(orderItemId, userId)`: `product_reviews`에서 `order_item_id = X`인 리뷰 조회(`maybeSingle`) — 있으면 수정 모드로 전환하기 위함. (같은 상품을 재구매한 다른 주문 건의 리뷰와는 무관하게 이 `order_item_id`에 한정된 리뷰만 본다.)
- `submitReview({ orderItemId, productId, userId, rating, title, content, images })`: insert, `order_item_id`/`is_verified_purchase: true` 고정, `images`는 `[{url}]` 배열. `idx_product_reviews_order_item_unique` 유니크 인덱스가 같은 구매 건에 대한 중복 작성을 막아준다. 트리거가 `products.review_count`/`average_rating`을 자동 갱신하므로 별도 처리 불필요.
- `updateReview(...)`.

### 2-2. `src/hooks/useProducts.ts` 수정 (Q&A 읽기/쓰기 보강)

- `fetchProductQna`에 `userId?: string` 파라미터 추가. 현재 `.eq("is_private", false)`만 걸려 있어 **본인이 쓴 비밀글이 본인에게도 안 보이는 공백**이 있다 → 로그인 시 `.or(\`is_private.eq.false,user_id.eq.${userId}\`)`로 교체 (RLS의 `product_qna_select_own` 정책과 일치시킴).
- `submitQna({ productId, userId, title, content, isPrivate })` 함수 추가: `product_qna`에 insert.

### 2-3. 이미지 업로드

리뷰 작성 모달 내부에서 `src/app/community/write/page.tsx:81-92`와 동일한 패턴으로 `totodo_pub_storage` 버킷에 `product-reviews/${userId}/${random}.${ext}` 경로로 업로드 후 `getPublicUrl` 결과를 로컬 state 배열에 누적, 제출 시 `images: string[]`로 훅에 전달.

## 3. 리뷰 작성 — `src/components/ProductReviewModal.tsx` (신규) + 구매내역 상세 페이지 연동

새 공용 모달 컴포넌트를 만들어 구매내역 상세에서 재사용한다. 쿠팡의 "구매한 상품" 리스트(개별 배송/구매 건마다 리뷰 작성하기 버튼)와 동일한 모델.

- Props: `orderItemId`, `productId`, `productName`, `onClose`, `onSubmitted`.
- 내부에서 `useMyProductReview(orderItemId, user.id)`로 `fetchMyReview` 호출 → 기존 리뷰가 있으면 `rating/title/content/images`를 프리필하고 "수정하기" 모드로 전환 (없으면 "등록하기").
- 폼 구성: `src/app/academy/[id]/reviews/page.tsx`의 `StarRating`(interactive) + 제목 인풋 + `textarea` 내용 + 다중 이미지 업로드 썸네일 영역(2-3 패턴).
- 제출 성공 시 `onSubmitted()` 콜백으로 부모에게 알림, 모달 닫힘.

### `src/app/settings/purchases/[id]/page.tsx` 수정

- "주문 상품" 섹션(line 569-615)의 각 `order.order_items` 행(=각 `order_item`)에 "리뷰 작성" 버튼 추가.
- 노출 조건: `order.status`가 `paid`/`shipped`/`delivered` 중 하나일 때만 (기존 `canRefund`/`eligibility` 계산과 같은 결의 판단 기준). `cancelled`/`pending` 주문에는 노출하지 않음.
- 라벨 정확도를 위해 주문 로딩 후 `product_reviews`를 `order_item_id IN (...이 주문의 item id들)`로 별도 조회해 `reviewedItemIds: Set<number>` state를 만든다 (쿠팡 스크린샷처럼 이미 쓴 항목은 "리뷰 작성하기" 대신 "리뷰 수정"/"숨기기"류 표시가 되도록). `useMyOrders`는 건드리지 않고 페이지 자체의 `useEffect`에서 가볍게 조회.
- 버튼 클릭 시 `reviewTarget: { orderItemId: item.id, productId: item.product_id, productName: item.product_name } | null` state를 세팅해 `ProductReviewModal` 오픈.
- 제출 성공(`onSubmitted`) 시 `reviewedItemIds`에 해당 `order_item_id` 추가.

## 4. 상품 상세 페이지 — `src/app/shop/[category]/[id]/page.tsx` 수정 (Q&A만 작성 가능, 리뷰는 조회 전용)

### 리뷰 탭 (line ~590)
- **작성 버튼(line 610-612)은 제거**하고, 대신 "구매하신 상품은 마이페이지 > 구매내역에서 리뷰를 작성할 수 있어요" 같은 안내 문구 정도만 남기거나 완전히 제거. 리스트 렌더링(line 617-648)은 그대로 유지 — 여기에 `review.images` 썸네일 표시만 추가.

### Q&A 탭 (line ~676)
- "문의하기" 버튼(line 708): `!user` → `LoginModal` 오픈. 로그인 시 인라인 폼 토글 (구매 여부 무관, 로그인만 필요).
- 폼: 제목 인풋 + `textarea` 내용 + "비밀글로 작성" 체크박스(`isPrivate`).
- Q&A 리스트에 `is_private` 항목은 이미 import돼 있으나 미사용인 `Lock` 아이콘(line 7)으로 뱃지 표시 — RLS 덕분에 리스트에 노출되는 비공개 항목은 항상 본인 것이므로 마스킹 없이 뱃지만 표시.
- 제출 성공 시 `fetchProductQna(productId, 1, user.id)` 재호출.

## 5. 어드민 훅 (신규)

## 검증 방법

1. `npm run dev` 실행 후 결제완료 상태의 주문이 있는 계정으로 `/settings/purchases/[id]` 접속 → 각 주문 상품에 "리뷰 작성" 버튼이 보이는지, 클릭 시 모달에서 별점+제목+내용+이미지 업로드 후 제출하면 정상 등록되는지 확인. 상품 상세 페이지(`/shop/[category]/[id]`)의 리뷰 탭에 반영되고 `average_rating`/`review_count`가 트리거로 자동 갱신되는지 확인.
2. `pending`/`cancelled` 상태 주문에는 리뷰 작성 버튼이 노출되지 않는지 확인.
3. 상품 상세 페이지 리뷰 탭에는 작성 버튼이 없고 조회만 되는지 확인.
4. 로그인 계정으로 상품 상세 Q&A "문의하기"에서 비밀글 체크 후 등록 → 본인에게는 보이고(Lock 뱃지), 다른 계정으로 접속 시 목록에서 사라지는지 확인.
5. 어드민 계정으로 `/admin/shop/products/[id]/reviews`에서 숨김 토글 시 상품 페이지 리뷰 탭에서 해당 리뷰가 사라지는지, 삭제 시 `review_count`가 감소하는지 확인.
6. `/admin/shop/products/[id]/qna`에서 답변 등록 후 상품 페이지 Q&A 탭에 "A." 답변이 노출되는지 확인.
7. `npm run lint` 통과 확인.
