# 상품 관리 2티어 메뉴 전환 + 카테고리별 상세 항목 스키마 관리

## Context

현재 어드민 "상품 관리"는 `/admin/shop/products` 단일 링크(1티어)다. 카테고리(도서/잡화)별 상세 입력 항목(저자, 출판사, 사이즈, 소재 등)은 `src/config/productMetaSchemas.ts`의 `CATEGORY_META_SCHEMAS` 객체에 slug(`books`, `goods`)로 하드코딩되어 있고, "이외" 카테고리는 DB에 존재하지도 않는다. 새 카테고리를 추가하려면 개발자가 이 TS 파일을 직접 고쳐야 한다.

목표는 "강의 관리"처럼 2티어 메뉴(상품 목록 / 상품 카테고리 목록)로 바꾸고, 카테고리 등록/수정 화면에서 관리자가 직접 "이 카테고리는 어떤 상세 항목을 가지는지"를 정의할 수 있게 해서, `ProductForm.tsx`/`productMetaSchemas.ts`의 하드코딩을 제거하는 것이다. 저장 데이터 형식(`product_details.type_meta` JSONB)은 이미 범용이므로 스키마 정의만 DB로 옮기면 된다.

## 설계 요약

- `product_categories` 테이블에 `field_schema JSONB` 컬럼 추가 — 카테고리별 상세 항목 정의를 `MetaField[]` 형태(`{key, label, type, placeholder?, options?, span?}`, 기존 `src/config/productMetaSchemas.ts`의 `MetaField` 타입 그대로 재사용)로 저장.
- 마이그레이션으로 기존 `books`/`goods`의 하드코딩된 스키마를 DB 데이터로 이관하고, `이외`(slug: `etc`) 카테고리를 새로 추가(빈 스키마 — 상세설명/비고는 이미 공통 필드라 그대로 둠).
- `product_categories`에 어드민 insert/update/delete RLS 정책 추가(현재 SELECT만 공개, 쓰기 정책 없음).
- `productMetaSchemas.ts`는 `CATEGORY_META_SCHEMAS` 삭제, `getMetaSchema(categoryFieldSchema, deliveryType)`로 시그니처 변경 — 카테고리 스키마는 DB에서 온 값을 그대로 받고, `DELIVERY_META_SCHEMAS`(digital_download/gifticon/coupon, 카테고리와 무관한 배송타입 축)는 그대로 유지(요청 범위 밖).
- 새 카테고리 CRUD 어드민 화면(`/admin/shop/categories`)에 필드 스키마 빌더 UI 추가 — 행 단위로 key/label/type/placeholder/options/span 편집, 추가/삭제/순서 이동.
- `AdminLayout.tsx`의 "상품 관리"를 `kind:"link"` → `kind:"group"`으로 변경, children: 상품 목록 / 상품 카테고리 목록.

## 변경 파일

### 1. `supabase/products.sql`
- `CREATE TABLE product_categories`에 `field_schema JSONB DEFAULT '[]'::jsonb` 컬럼 추가.
- RLS 섹션에 `product_categories_insert_admin` / `_update_admin` / `_delete_admin` 정책 추가 (기존 `products_*_admin` 패턴과 동일하게 `public.is_admin()` 사용).
- 파일 하단 마이그레이션 섹션에 기존 운영 DB용 블록 추가:
  - `ALTER TABLE product_categories ADD COLUMN IF NOT EXISTS field_schema JSONB DEFAULT '[]'::jsonb;`
  - `DROP POLICY IF EXISTS ...; CREATE POLICY ...` 3종 (idempotent하게)
  - `UPDATE product_categories SET field_schema = '[...]'::jsonb WHERE slug = 'books'` — 현재 `CATEGORY_META_SCHEMAS.books` 내용을 JSON으로 그대로 이관 (isbn, author, publisher, publish_date, book_type, print_color, age_limit, page_count, author_introduction, table_of_contents + shipping_fee)
  - 동일하게 `goods` 이관 (size, material, published_by, distributor + shipping_fee)
  - `INSERT INTO product_categories (name, slug, description, field_schema) VALUES ('이외', 'etc', '도서/잡화 외 상품', '[]'::jsonb) ON CONFLICT (slug) DO NOTHING;`

### 2. `src/config/productMetaSchemas.ts`
- `CATEGORY_META_SCHEMAS` 상수 제거.
- `getMetaSchema` 시그니처 변경:
  ```ts
  export function getMetaSchema(
    categoryFieldSchema: MetaField[] | null | undefined,
    deliveryType: string
  ): MetaField[] {
    if (categoryFieldSchema && categoryFieldSchema.length > 0) return categoryFieldSchema;
    return DELIVERY_META_SCHEMAS[deliveryType] ?? [];
  }
  ```
- `MetaField`, `MetaFieldType`, `SHIPPING_FEE`, `DELIVERY_META_SCHEMAS`는 그대로 유지(SHIPPING_FEE는 카테고리 스키마 빌더에서 "많이 쓰는 필드" 프리셋으로 재사용 가능하도록 export 유지).

### 3. `src/hooks/useAdminProducts.ts`
- `ProductCategory` 인터페이스에 `field_schema: MetaField[]` 추가 (import `MetaField` from `@/config/productMetaSchemas`).
- `fetchCategories()`의 select 컬럼에 `field_schema` 추가.

### 4. `src/app/admin/shop/products/write/page.tsx`, `src/app/admin/shop/products/[id]/edit/page.tsx`
- 인라인 `supabase.from("product_categories").select("id, name, slug")` → `"id, name, slug, field_schema"`로 확장.

### 5. `src/app/admin/shop/products/_components/ProductForm.tsx`
- `const metaSchema = getMetaSchema(selectedCategory?.slug ?? "", form.delivery_type);` → `getMetaSchema(selectedCategory?.field_schema, form.delivery_type);`로 변경. 나머지 렌더링 로직(`MetaFields` 등)은 그대로.

### 6. `src/hooks/useAdminProductCategories.ts` (신규)
`useAdminCodes.ts` 패턴 참고. 제공 기능:
- `fetchCategories()` — 전체 카테고리 목록(`id, name, slug, description, field_schema, created_at`) + 카테고리별 상품 개수(간단히 별도 count 쿼리 또는 생략 가능 — 목록엔 상품수 없이 이름/슬러그/필드수만 표시)
- `createCategory({ name, slug, description, field_schema })`
- `updateCategory(id, updates)`
- `deleteCategory(id)` — 삭제 전 해당 category_id를 가진 `products` row 존재 여부를 확인해서, 존재하면 삭제를 막고 실패 사유를 반환 (FK가 `ON DELETE CASCADE`라서 그냥 삭제하면 상품이 통째로 날아감 — 반드시 가드 필요)

### 7. `src/app/admin/shop/categories/page.tsx` (신규) — 카테고리 목록
`src/app/admin/codes/page.tsx` / `AdminTable` 패턴 참고. 컬럼: 이름 / slug / 상세 항목 수 / 관리(수정 이동, 삭제). 상단 `AdminPageHeader`에 `action={{ label: "새 카테고리 등록", href: "/admin/shop/categories/write" }}`.

### 8. `src/app/admin/shop/categories/_components/CategoryForm.tsx` (신규) — 등록/수정 공용 폼
`ProductForm.tsx`의 레이아웃 톤(section/FieldRow 스타일) 참고. 구성:
- 기본 정보: 이름, slug(신규 등록시만 수정 가능 — 상품에서 참조 중인 slug를 바꾸면 `getMetaSchema` 폴백 로직 등과 꼬일 수 있어 수정 모드에서는 읽기전용 처리), 설명
- 상세 항목 스키마 빌더(같은 파일 내 하위 컴포넌트 `FieldSchemaEditor`): `field_schema` 배열을 행 리스트로 렌더링, 각 행에 key(영문 snake_case, 카테고리 내 유니크 검증)/label/type(select: text/number/select/textarea/richtext/date)/placeholder/options(type=select일 때만, 콤마 구분 입력)/span(half/full) 입력, 행 추가/삭제/위아래 순서 이동 버튼.
- 저장 시 key 중복·빈 값 검증 후 `createCategory`/`updateCategory` 호출, 완료 후 `/admin/shop/categories`로 이동.

### 9. `src/app/admin/shop/categories/write/page.tsx`, `src/app/admin/shop/categories/[id]/edit/page.tsx` (신규)
`products/write`, `products/[id]/edit` 페이지와 동일한 얇은 wrapper 패턴(auth 체크 + 데이터 로드 + `<CategoryForm mode=... />`).

### 10. `src/components/AdminLayout.tsx`
`navItems`의 상품 관리 항목을:
```ts
{
  kind: "group",
  label: "상품 관리",
  icon: ShoppingBag,
  basePath: "/admin/shop",
  children: [
    { href: "/admin/shop/products", label: "상품 목록" },
    { href: "/admin/shop/categories", label: "상품 카테고리 목록" },
  ],
},
```
로 교체 (강의 관리 그룹과 동일한 구조라 `openGroups`/활성 상태 로직은 수정 불필요).

## 검증

- `npm run lint` 통과 확인.
- 개발 서버 기동 후 브라우저로:
  1. 사이드바에서 "상품 관리"가 강의 관리처럼 펼침/접힘 그룹으로 보이는지, "상품 목록"·"상품 카테고리 목록" 두 하위 메뉴로 이동되는지 확인.
  2. `/admin/shop/categories`에서 도서/잡화/이외 3개 카테고리가 보이는지, 새 카테고리 등록이 되는지, 상세 항목(필드) 추가/수정/삭제가 되는지 확인.
  3. 상품이 존재하는 카테고리를 삭제 시도 → 차단되는지 확인.
  4. `/admin/shop/products/write`에서 카테고리를 "도서"로 선택했을 때 기존과 동일하게 저자/출판사/목차 등 필드가 뜨는지(마이그레이션된 데이터로 하드코딩 시절과 동일 UI 재현), "이외"로 선택했을 때 카테고리 전용 필드 없이 상세설명/비고만 남는지 확인.
  5. 카테고리 관리 화면에서 "도서" 카테고리에 새 필드를 추가한 뒤, 상품 등록 폼에 즉시 반영되는지 확인 — 하드코딩 제거가 실제로 동작함을 증명하는 핵심 시나리오.
