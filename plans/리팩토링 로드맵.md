# totodo-fe 전체 리팩토링 로드맵

## Context

기능 구현이 대부분 끝난 시점에서, 그동안 빠르게 기능을 쌓아오며 생긴 구조적 부채를 단계적으로 정리하기 위한 리팩토링이다. 조사 결과 크게 세 가지 층위의 부채가 확인됐다:

1. **UI 레이어** — `src/components/admin/{atoms,molecules,organisms}/`에 이미 잘 구축된 atomic 패턴이 있지만 관리자 페이지 28곳에서만 쓰이고, 일반 사용자 페이지(`shop`/`academy`/`faq`/`community`/`settings`)에서는 전혀 재사용되지 않아 모달 6종·스피너 2종+인라인 37곳·RichTextEditor 3벌 등 반복 구현이 쌓여 있다.
2. **상수/코드 레이어** — `codes` 공통 코드 테이블과 관리 화면은 이미 있지만 자기 자신 외에는 아무도 소비하지 않고, 주문상태·배송타입·환불상태·쿠폰타입 등의 라벨이 파일마다 다른 이름·다른 값 집합으로 중복 정의돼 있다(실제로 배송추적상태의 `"shipped"`와 주문상태의 `"shipped"`가 서로 다른 의미로 충돌하는 사례까지 확인됨).
3. **API/도메인 로직 레이어** — 서비스 레이어 없이 ~40개 훅이 Supabase를 직접 호출하고, `requireAdmin()`이 6곳에 복붙돼 있으며, 결제/배송 쪽에는 실제로 살아있는 버그 2건이 발견됐다.

사용자가 제안한 4-phase 순서(프론트 공통화 → 상수/공통코드화 → CRUD 공통화 → 결제/배송 고급화)는 **블라스트 레이디어스가 작은 것부터 큰 것 순**이라는 정석과 일치하므로 그대로 유지한다. 다만 조사 과정에서 발견한 활성 버그 2건은 사용자 결정에 따라 **Phase 0으로 분리해 최우선 처리**하고, 그 외 세부 스코프는 아래 질의응답으로 확정했다:

- 활성 버그 2건 → **Phase 0으로 즉시 별도 처리** (리팩토링 로드맵과 분리)
- Phase 1 컴포넌트 공유 전략 → **패턴만 통일, 세트는 분리** (admin/사용자 컴포넌트 파일을 각각 유지하되 이름·구조·prop 셰이프를 통일)
- 테스트 자동화 도입 → **이번 로드맵 범위 밖** (Vitest 등 신규 도입 없이 수동 회귀 체크리스트로 리스크 관리)
- Phase 2 DB 데이터 이관 → **포함** (라벨 통일뿐 아니라 book_type/board_category 등 실제 저장값도 이관)

이 문서는 각 Phase의 실행 순서와 근거를 정리한 로드맵이며, 실제 구현은 Phase별로 별도 세션에서 진행한다.

## Phase 0 — 활성 버그 긴급 대응 (리팩토링 착수 전, 최우선)

### 0-1. 배송비 정책 불일치 해소
- **현상**: `src/app/shop/checkout/page.tsx:119`의 실제 로직은 `physicalSubtotal < 30000`이면 배송비 3,000원 부과(3만원↑ 무료)인데, 고객 안내 문구 `src/config/deliveryPolicyInfo.ts:13`은 "50,000원 이상 구매 시 무료배송"이라 명시돼 있어 실제 계산과 안내가 다르다.
- **선행 조건**: `plans/digital_physical_item.md`(기존 설계 문서)에 무료배송 기준 금액에 대한 근거가 없으므로, 3만원/5만원 중 무엇이 의도된 정책인지 **사용자(PO) 확인 필요**.
- **처리**: 확정된 값으로 로직과 문구 중 어긋난 쪽만 일치시킨다. 이번 단계에서는 값을 config/codes로 승격하지 않고 숫자만 맞춘다 — 승격 여부는 Phase 4 7단계에서 운영자가 자주 바꿀 값인지 확인 후 결정.

### 0-2. `shipping_tracking` 쓰기 경로 최소 추가
- **현상**: `supabase/order.sql`에 `shipping_tracking` 테이블(`delivered_at` 등)이 정의돼 있고, `src/lib/fulfillment/cancelPolicy.ts`의 `canRefundOrderByDeliveryState()`/`canRefundItem()`은 실물 상품 환불 조건으로 `order.status === "delivered" && delivered_at`(7일 이내)을 요구한다. 그런데 `.from("shipping_tracking")`은 `payment/cancel/route.ts`, `orders/refund-request/route.ts` 두 곳에서 **SELECT만** 하고 INSERT/UPDATE는 코드베이스 어디에도 없다. `src/hooks/useAdminOrders.ts`의 `updateStatus()`(263-277행)도 `orders.status`만 갱신할 뿐이다. 즉 **관리자가 주문을 "배송완료"로 바꿔도 `delivered_at`이 영구히 null로 남아 실물 상품의 7일 환불 신청이 사실상 항상 불가능**한 상태다.
- **처리**:
  1. `useAdminOrders.ts`에 `updateShippingTracking()` 함수 추가 — 상태를 `shipped`로 바꿀 때 `shipped_at`, `delivered`로 바꿀 때 `delivered_at`을 함께 기록(row가 없으면 upsert).
  2. 관리자 주문 상세/목록에 운송장번호·택배사 입력 최소 폼 1개 추가. 이 시점엔 Phase 1 컴포넌트가 아직 없으므로 기존 admin 스타일로 최소 구현하고, Phase 4에서 정식 컴포넌트로 교체할 것을 전제로 과설계하지 않는다.
  3. 배포 후 이미 `delivered` 상태인데 `delivered_at`이 null인 과거 주문을 소급 처리할지 운영팀과 확인(백필 여부 결정 필요).
- **주의**: 이 항목은 "긴급 기능 보강"이며 confirm/cancel 플로우와의 정식 통합·상태값 이름 정리는 Phase 4/Phase 2에서 다시 다룬다.

## Phase 1 — 프론트엔드 공통/표준화

### 방향: 패턴 통일 + 세트 분리 (사용자 확정)

admin(라이트 테마, `#cc785c` 계열, inline hex)과 사용자 화면(다크 테마, `--color-brand-500: #a200cb`, Tailwind 클래스, `globals.css:19`에서 확인)은 색상 체계가 반대이므로, 컴포넌트 파일을 완전히 공유하지 않는다. 대신:

- `src/components/ui/{atoms,molecules,organisms}/`를 신설해 admin과 **동일한 이름·디렉토리 계층·prop 셰이프**를 따르되 다크 테마로 별도 구현한다.
- 단, 페이지네이션 계산이나 검색 debounce처럼 **순수 상태 로직**은 헤드리스 훅(`usePagination`, `useDebouncedSearch` 등)으로 한 번만 만들어 admin/user 양쪽 프레젠테이션 컴포넌트가 공유한다. 마크업만 두 벌이고 로직은 한 벌이라는 원칙.
- 예외: RichTextEditor는 admin/user 모두 "텍스트 편집"이라는 동일 도메인이고 테마 차이의 실익이 적어 완전 공유한다.

### 실행 순서 (안전하고 재사용 빈도 높은 것부터)

1. **로딩 스피너 통합** — `LoadingSpinner.tsx`, `PageLoading.tsx`, 37개 파일의 인라인 `animate-spin` div를 `ui/atoms/Spinner`(admin `Spinner.tsx`와 동일 API: `size`/`color`)로 흡수. 순수 시각적 교체라 가장 안전한 첫 스텝.
2. **공통 `Modal` molecule 신설** — `LoginModal`, `AddressFormModal`, `AddressSearchModal`, `DeleteAccountModal`, `VerifyPasswordModal`, `ProductReviewModal` 6곳이 반복하는 `fixed inset-0 z-50 flex items-center justify-center bg-black/70` 오버레이/카드 뼈대만 `ui/molecules/Modal`로 추출. **내부 폼 검증/제출 로직은 건드리지 않고 각 모달에 그대로 둔다** — 마크업 중복 제거가 목표이지 비즈니스 로직 리팩토링은 Phase 3 영역이다.
3. **RichTextEditor 3벌 통합** — `src/components/RichTextEditor.tsx`, `src/components/admin/blocks/RichTextEditor.tsx`, `src/components/admin/AdminRichTextEditor.tsx`의 TipTap extension 설정 차이를 표로 정리한 뒤 통합, 차이 나는 기능은 `features` prop으로 옵션화.
4. **리스트+검색+페이지네이션 마크업 통일** — `faq/page.tsx`, `community/page.tsx`(faq 페이지네이션 JSX를 그대로 복붙한 상태), `shop/[category]/page.tsx`, `settings/purchases/page.tsx`(로컬 Pagination 재정의)에 `ui/molecules/SearchBar`, `ui/molecules/Pagination`, `ui/molecules/FilterTabs` 적용. 데이터 패칭 로직(`.range()` 골격) 통합은 Phase 3으로 남기고 이번엔 마크업만 교체.
5. **카드형 컴포넌트 정리** — `shop/wishlist/page.tsx`가 재구현한 상품 카드를 기존 `src/components/shop/ProductCard.tsx`로 교체(필요 시 `variant` prop 추가).
6. **범용 Button/Input atom 신설** — `ui/atoms/Button`, `ui/atoms/Input` 신설. 기존 페이지 전면 치환은 하지 않고 "신규 작성 시 표준"으로 지정 후 점진 교체.
7. **CLAUDE.md에 "User-facing Pages — Atomic Design Pattern" 섹션 추가** — 기존 "Admin Pages — Atomic Design Pattern"과 대칭되는 문서를 남겨 다크 테마 팔레트(`brand-500 = #a200cb` 등)와 `ui/` 임포트 규칙을 명문화. 이 문서화가 없으면 Phase 4에서 만들 운송장 입력 UI 등이 다시 흩어진다.

**마이그레이션 전략**: 컴포넌트 신설 PR과 기존 페이지 전체 교체 PR을 분리하고, 시각 회귀 테스트 도구가 없으므로 교체 PR마다 브라우저로 직접 확인하는 절차를 명시한다.

## Phase 2 — 상수/공통코드(COMMON_CODE)화

### 핵심 결정: `codes` 완전 FK 전환은 하지 않는다

`codes` 테이블은 `UNIQUE(group_code, code)` 복합키라 단일 컬럼 FK가 불가능하다(codes.sql 주석에 명시된 기존 제약). 세 대안을 검토한 결과:
- **완전 FK 전환**(컬럼을 `code_id INTEGER REFERENCES codes(id)`로 교체) → 이를 참조하는 모든 훅·API·필터링(`.eq("delivery_type","physical")` 등)이 깨져 Phase 2에 Phase 4급 리스크를 끌어들임 → **비권장**
- **트리거 검증 유지 + 확대**(기존 `validate_delivery_type()` 패턴을 다른 그룹에도 복제) → 스키마 변경이 트리거 함수 추가뿐이라 리스크 작음 → **채택**

### 2-A. 인벤토리 및 원칙 문서화
1. 조사에서 확인된 중복 상태값 목록(주문상태/배송추적상태/delivery_type/환불상태/discount_type/미션 제출상태/book_type/BOARD_CATEGORY)을 "필드명 / 저장 위치 / 중복 파일 / codes 존재 여부 / 값 인코딩 일치 여부" 표로 `plans/refactoring-constants.md`에 문서화(기존 `plans/coupon-design.md` 컨벤션 따름).
2. 원칙 확정: **운영자가 값 자체를 추가/수정해야 하는 마스터데이터(delivery_type, book_type 등)는 `codes` 테이블로, 상태 전이 로직이 코드에 강결합된 상태값(order.status처럼 순서 자체가 로직인 것)은 TS union 타입으로 유지**.

### 2-B. 라벨/타입 레이어 통합 (저리스크)
- **`order.status` 통합**: `useMyOrders.ts`의 `MyOrderStatus`(pending/cancelled 누락)와 `useAdminOrders.ts`의 `OrderStatus`를 `src/types/order.ts`(신규, `src/types/coupon.ts`와 동일 컨벤션)로 통합. `settings/purchases/page.tsx`와 `[id]/page.tsx`에 완전히 동일하게 복붙된 `STATUS_LABEL`/`STATUS_CLASS`/`REFUND_STATUS_LABEL`도 `src/config/orderLabels.ts`로 이관.
- **배송추적상태 `"shipped"` 충돌 해소**: `TRACKING_STATUS_LABEL`(`preparing/shipped/in_transit/delivered`)의 `shipped`(발송완료)와 order.status의 `shipped`(배송중)가 같은 문자열로 다른 의미 — 이건 DB 컬럼 값 자체를 바꾸는 작업(예: tracking 쪽을 `dispatched`로 개명)이라 UPDATE 마이그레이션이 필요하므로 2-C 취급(백업 후 진행).
- **delivery_type 라벨 통합**: `ProductCard.tsx`, `shop/[category]/[id]/page.tsx`, `settings/purchases/[id]/page.tsx`, `admin/orders/page.tsx`, `admin/shop/products/page.tsx` 등에 이름만 다르게(`DELIVERY_TYPE_LABELS`/`DELIVERY_TYPE_LABEL`/`DELIVERY_LABELS`) 중복된 동일 객체를, `codes` 테이블을 런타임 조회하는 신규 `useCodes` 훅으로 대체. 이때 `useAdminCodes.ts`의 `CODE_GROUPS`에 `BOARD_CATEGORY`가 빠진 것도 함께 수정.
- **쿠폰 discount_type 라벨 통합**: `admin/coupons/page.tsx`, `admin/coupons/[id]/page.tsx`, `ProductForm.tsx`, `settings/my-coupons/page.tsx`, `shop/checkout/page.tsx`의 표시용 중복을 `src/config/couponLabels.ts`의 `formatCouponDiscount(coupon)` 함수로 통합(`src/types/coupon.ts`의 기존 `DiscountType` 재사용). **주의**: `api/coupons/validate/route.ts`의 실제 할인 금액 계산 로직은 라벨이 아니라 비즈니스 로직이므로 이번 단계에서 손대지 않고 Phase 3/4로 남긴다.
- **미션 제출 상태 라벨 통일**: 사용자 화면("제출 완료")과 관리자 화면("제출됨") 문구 통일.

### 2-C. DB 데이터 이관 (사용자 확정: 포함)
- **`shipping_tracking.status` 값 개명**: `shipped` → `dispatched` 등으로 변경해 order.status와의 충돌 제거. 기존 데이터 UPDATE + CHECK 제약 추가.
- **`BOARD_CATEGORY` 이관**: codes 시드값(`notice/question/general/column`)과 `lecture_board_posts.category`의 실제 저장값(한글 자유 텍스트, CHECK 제약 없음)을 매핑해 기존 데이터를 영문 code로 UPDATE 후, `validate_delivery_type()`과 동일한 트리거 추가.
- **`book_type` 이관**: `codes.code`(영문 `paperback`)와 `ProductForm.tsx`의 실제 저장값(한글 "종이책")이 다르므로 기존 데이터 이관이 필요. 단, 저장 로직 자체(`ProductForm.tsx`)를 codes 참조 방식으로 바꾸는 것은 폼 리팩토링과 경계가 겹치므로 **트리거 검증 활성화는 Phase 3에서 `ProductForm.tsx`를 다룰 때 함께 처리**하고, 2-C에서는 데이터 이관과 라벨 통일까지만 완료한다.
- 모든 데이터 UPDATE는 운영 DB 백업 후 진행하고, 이관 전후 row count/distinct value로 정합성 검증한다.

## Phase 3 — 단순 CRUD API 로직 개선/공통화

### 방향: 얇은 쿼리 헬퍼(순수 함수) — 완전한 리포지토리 패턴은 비권장

이미 `src/lib/fulfillment/cancelPolicy.ts`, `digitalFulfillment.ts`가 "순수 함수 + 명시적 타입 + 서버/클라이언트 겸용"이라는 좋은 선례를 만들어뒀으므로 이 스타일을 그대로 확장한다. 클래스 기반 리포지토리는 이 프로젝트 규모 대비 과설계.

### 실행 순서 (저리스크 → 고리스크)

1. **`requireAdmin()` 추출 (최우선)** — `admin/coupons/route.ts`, `[id]/route.ts`, `admin/ebook/upload-url/route.ts`, `admin/ebook/delete/route.ts`, `admin/gifticon/reissue/route.ts` 5개 파일에 복붙된 함수와 `admin/coupons/[id]/issue/route.ts`에 인라인된 동일 로직을 `src/lib/auth/requireAdmin.ts`로 추출. 반환 타입(`{user,admin}` vs `{admin}`)도 통일. 순수 이동이라 회귀 확인이 "각 admin API가 여전히 401/403/정상 응답하는지" 정도로 가벼움 — 이후 스텝들의 검증 패턴 기준점.
2. **`auth/delete-account/route.ts`의 `@supabase/supabase-js` 직접 import 정리** — 기존 `src/utils/supabase/admin.ts`의 `createAdminClient()`로 교체. 계정 삭제라는 민감 액션이므로 교체 후 FK cascade 정상 동작을 수동 QA.
3. **페이지네이션 쿼리 헬퍼 추출** — `src/lib/query/paginate.ts`에 `.range()` + `{count:"exact"}` + `ilike` 검색 골격을 감싸는 헬퍼 작성. `useFaqs`부터 시범 적용 후 `useAdminProducts`/`useAdminReviews`/`useAdminUsers`/`useAdminOrders`/`useAdminCoupons`/`useLectureReviews` 등 14개 훅에 순차 이관. PAGE_SIZE(10/12/15로 제각각)는 화면별 의도된 값인지 먼저 확인 후 통일 여부 결정.
4. **삭제/토글 CRUD 경로 수렴** — 훅도 안 거치고 페이지 컴포넌트가 직접 Supabase를 호출하는 `admin/faq/page.tsx`의 `handleDelete`부터 훅으로 승격. "RLS만으로 충분한 단순 CRUD는 훅에서 client 직접 호출 유지 / service-role이나 부가 로직(알림 등)이 필요한 것만 API route로 승격"이라는 기준을 문서화하고 나머지를 정리. `useMyCoupons`/`useCouponValidate`의 "조회는 client 직접, 쓰기는 API route" 패턴은 흔한 절충안이므로 무리하게 전부 API route로 옮기지 않는다.
5. **API 성공 응답 envelope 표준화** — 기존 응답 형태가 제각각(`{success:true}` / `{coupon}` / `{coupons,total,page,limit}`)인 것을, 기존 라우트를 전부 한 번에 바꾸지 않고 **신규/리팩토링 대상 route부터 신규 컨벤션 적용, 기존은 실제로 손댈 일이 생겼을 때 이관**.
6. **`book_type` 저장 로직 정리** — Phase 2-C에서 보류한 `ProductForm.tsx`의 `DELIVERY_OPTIONS` 하드코딩과 `book_type` 저장 로직을 codes 참조 방식으로 전환하고, 이 시점에 트리거 검증도 활성화.

## Phase 4 — 결제/배송 등 고급 로직 개선/공통화

테스트 자동화 도입은 범위 밖으로 확정했으므로, 이미 있는 순수 함수 분리 패턴(`cancelPolicy.ts`)을 더 확장하는 방향으로 리스크를 낮추고 **수동 회귀 체크리스트**로 안전망을 대신한다.

### 실행 순서

1. **동작 불변 리팩토링 우선, 동작 변경은 별도 커밋** — `payment/confirm/route.ts`와 `payment/free-confirm/route.ts`가 공유하는 "주문 조회→상태 확인→쿠폰 재검증→디지털 상품 사전검증→paid 갱신→쿠폰 사용 처리→디지털 이행 생성→장바구니 비우기" 8단계를 `src/lib/fulfillment/orderConfirm.ts`로 추출하고, Toss 호출 부분만 `chargeStep?: () => Promise<TossResult>` 콜백으로 주입(free-confirm은 생략). 과거 "free-confirm이 로직을 빠뜨렸던" 드리프트 버그가 구조적으로 재발하지 않게 하는 효과. **이 추출과 "재고 차감·shipping_tracking 정식화 같은 새 로직 추가"는 반드시 별도 PR로 분리** — 한 PR에 구조 변경과 동작 변경이 섞이면 문제 발생 시 원인 추적이 어려움.
2. **실물 상품 재고 차감 신규 구현** — 기프티콘의 `claim_gifticon_code` RPC(원자적 차감 + 소진 시 부분환불)와 동일 패턴으로 `UPDATE products SET stock = stock - 1 WHERE id = ? AND stock > 0 RETURNING stock` 형태의 원자적 Postgres 함수를 만들고 `payment/confirm` 트랜잭션 내부에서 실행해 레이스 컨디션을 막는다.
3. **`shipping_tracking` 정식화** — Phase 0의 최소 구현을 Phase 1에서 만든 `ui`/`admin` molecules로 정식 컴포넌트화하고, Phase 2-C에서 개명한 상태값(`dispatched` 등) 정합성을 재확인.
4. **주문 생성 로직 통합** — `useOrders.ts`(`createOrder`)와 `useLectureOrder.ts`(`createLectureOrder`)의 `generateOrderNumber()` + "주문 insert → order_items insert → 실패 시 롤백" 중복을 공유 함수로 통합. 이 과정에서 **`useLectureOrder.ts`가 `order_type`을 아예 insert 객체에 포함하지 않아 DB 기본값인 `'physical'`로 잘못 저장되는 버그**(직접 확인 완료 — `useOrders.ts:64,82`는 `order_type`을 physical/digital/mixed로 파생해 명시적으로 insert하는데 `useLectureOrder.ts:30-41`엔 이 필드 자체가 없음)를 함께 수정한다.
5. **배송비 정책값 설정화 여부 검토** — Phase 0에서 값만 맞췄던 무료배송 기준액을 운영팀이 프로모션으로 자주 바꿀 값인지 확인 후, 그렇다면 `codes` 또는 별도 config로 승격.
6. **안전 배포 완화책** — feature flag 인프라가 없으므로 신규 도입 대신, 재고 검증처럼 리스크 큰 변경은 "처음엔 위반 시에도 차단하지 않고 로그만 남기는 soft-launch 모드로 배포 → 며칠 관찰 후 hard 모드 전환" 방식과 "트래픽 적은 시간대 배포 + 배포 직후 관리자가 테스트 결제 1건 직접 실행"으로 리스크를 낮춘다.
7. **수동 회귀 체크리스트 작성 및 각 배포 직후 실행** (테스트 자동화가 범위 밖이므로 필수 산출물):
   - 정상 카드결제(실물/디지털/혼합 장바구니 각각)
   - 0원 확정(쿠폰 100% 할인)
   - 결제 도중 쿠폰 만료
   - 기프티콘 재고 소진 시 부분환불
   - 실물 배송완료 처리 후 7일 이내/이후 환불 신청 각각 (Phase 0-2가 실제로 동작하는지 확인하는 핵심 케이스)
   - 강의 진척률 10% 초과 후 취소 시도
   - 재고 0인 상품 동시 주문(레이스 컨디션)
   - 강의 주문 생성 시 `order_type`이 이제 올바르게 저장되는지

## Phase 간 의존관계

- **Phase 0은 항상 최우선** — 나머지 모든 Phase보다 먼저 하거나 최소한 Phase 1과 병행.
- **Phase 1 ↔ Phase 2는 약한 상호 의존** — Phase 2가 만들 라벨/색상 매핑을 Phase 1의 `Badge`/`FilterTabs`가 prop으로 소비하지만, Phase 1을 "라벨을 prop으로 받는" 인터페이스로 설계해두면 완전한 선행 요구는 아니다. 두 Phase를 병행해도 무방.
- **Phase 1 → Phase 4 약한 선행** — Phase 4의 운송장 입력 폼 등 신규 admin UI를 Phase 1 산출물(molecules)로 바로 조립 가능해 순서를 지키면 이득이 있다(강제는 아님).
- **Phase 2-B/2-C → Phase 3 선행 권장** — 상태값 이름(예: tracking `shipped`→`dispatched`)이 Phase 2에서 확정돼야 Phase 3의 공통 쿼리 헬퍼/타입이 이중 작업 없이 그 이름을 바로 채택할 수 있다.
- **Phase 3 → Phase 4 강한 선행 권장** — Phase 4의 주문 생성/결제 확정 통합은 Phase 3에서 확립할 "서비스 레이어 원칙"(어디까지 `lib` 함수로 뺄지, 응답 envelope 등)을 따라야 일관성이 생긴다. 단, `lib/fulfillment`가 이미 어느 정도 분리돼 있어 착수 자체가 Phase 3을 기다릴 필요는 없다 — "먼저 끝내야 일관성 있게 끝난다"는 의존.

## 검증 방법 (테스트 스위트 없음 — 수동 검증 중심)

- **공통**: 각 Phase 완료 시 `npm run lint` 통과 확인.
- **Phase 0**: `npm run dev`로 체크아웃 페이지에서 3만원 경계값 안팎 주문 시 배송비 표시 확인, 관리자 주문 상세에서 배송상태를 shipped→delivered로 바꾼 뒤 `shipping_tracking` row에 `delivered_at`이 채워지는지 Supabase 대시보드로 직접 확인.
- **Phase 1**: 교체한 각 페이지(`/faq`, `/community`, `/shop/[category]`, `/shop/wishlist`, `/settings/purchases`, 로그인/주소/비밀번호 확인 모달 등)를 브라우저로 직접 열어 다크 테마 스타일이 깨지지 않았는지, admin 쪽(`/admin/**`)은 이번 변경으로 전혀 영향받지 않았는지 확인.
- **Phase 2**: `/admin/codes`에서 모든 그룹(BOARD_CATEGORY 포함)이 정상 노출되는지, 데이터 이관 후 이관 전/후 row count와 distinct value가 일치하는지 SQL로 확인, 라벨이 바뀐 화면들(주문 상태, 배송추적, 쿠폰 등) 육안 확인.
- **Phase 3**: 각 admin API route를 관리자/비관리자 계정으로 호출해 401/403/200이 기존과 동일하게 동작하는지, admin CRUD 화면(삭제/토글)이 정상 동작하는지 확인.
- **Phase 4**: 위에 명시한 수동 회귀 체크리스트 전체를 각 배포 직후 실행. 특히 실물 상품 7일 환불 신청 케이스는 Phase 0-2와 Phase 4가 함께 만든 기능이므로 end-to-end로 반드시 확인.
